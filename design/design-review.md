## 1. 项目基本情况

### 1.1 产品定位

| 维度         | 内容                                                                                           |
| ------------ | ---------------------------------------------------------------------------------------------- |
| **产品名**   | Snow Cues（雪提示）                                                                            |
| **版本**     | v2.1 → v2.2                                                                                    |
| **产品类型** | 本地优先密码派生与加密存储系统                                                                 |
| **技术定位** | 纯前端 PWA，无后端、无账号系统、无云同步                                                       |
| **分发方式** | Cloudflare Pages 静态托管（仅分发前端代码，不承载业务数据）                                    |
| **技术栈**   | Vite + React 19 + TypeScript                                                                   |
| **目标用户** | 注重隐私的技术用户（以中文为主要语言）                                                         |
| **运行环境** | 桌面/移动端浏览器，需要 HTTPS 安全上下文或 localhost，依赖 WebCrypto 和 File System Access API |

### 1.2 核心功能模块

```
┌──────────────────────────────────────────────────┐
│                  Snow Cues v2.2                   │
│                                                   │
│  session-manager  │  主密码 → WebCrypto key      │
│  rule-registry    │  静态 pure function 规则注册表│
│  crypto-engine    │  规则执行、AES-GCM 加解密     │
│  recovery-aid     │  关键密钥记忆提示加密         │
│  storage-data     │  storageData 文件读写         │
│  space            │  空间 policy、克隆、迁移      │
│  ui               │  React UI 层（页面+组件）     │
└──────────────────────────────────────────────────┘
```

### 1.3 UI 架构

- **布局模式**：左侧垂直导航轨道 (220px) + 中间内容区 + 右侧悬浮指引抽屉
- **路由方式**：hash 路由 (`#/space`, `#/rules`, `#/groups`, `#/passwords`, `#/tools`)
- **组件分层**：基础设计系统组件 (`src/ui/design-system/`) → 业务组件 (`src/ui/components/`) → 页面 (`src/ui/pages/`)
- **样式方案**：纯手写 CSS + CSS Custom Properties，单文件 `src/styles.css`（1910 行），无 UI 组件库
- **响应式**：单断点 960px

## 3. 现存问题诊断

### 3.1 核心理念冲突：品牌名与视觉语言的背离

| 品牌元素                | 暗示感受             | 实际视觉         | 矛盾                |
| ----------------------- | -------------------- | ---------------- | ------------------- |
| **"Snow"**（雪）        | 冷、净、白、冰、清澈 | 暖奶油黄底色     | ❌ 暖 vs 冷         |
| **"Cues"**（提示/线索） | 精确、锐利、逻辑     | 1.5–2rem 大圆角  | ❌ 锐利 vs 柔和     |
| **密码安全工具**        | 可信、冷静、坚固     | 陶土橙、有机质感 | ❌ 安全 vs 手工艺感 |

**一句话概括**：当前设计在说"手工陶艺坊"或"精品咖啡馆"，但产品叫 "Snow Cues"，做的是"密码安全系统"。

### 3.2 色彩心理学问题

| 颜色                 | 当前使用                       | 常规心理联想           | 对安全产品的影响         |
| -------------------- | ------------------------------ | ---------------------- | ------------------------ |
| **陶土橙** `#d25636` | 主色（按钮、焦点环、导航指示） | 温暖、有机、手工、食欲 | 传达"手艺人"而非"守护者" |
| **暖奶油** `#f3ece1` | 页面背景                       | 复古、纸张、陈旧       | 长时间阅读有"泛黄纸张"感 |
| **暖卡其** `#e2d9cc` | 卡片/面板                      | 自然、柔和、低科技     | 不够现代、不够精确       |
| **暗橄榄** `#232018` | 暗色模式底色                   | 泥土、温暖暗调         | 泥泞感，对比度不够干脆   |

在色彩心理学中，**冷色（蓝、青）天然传达专业、信任、安全、技术**，暖色（橙、棕、黄）传达温暖、有机、手工。密码安全产品使用暖橙色作为唯一主色是一个**品类的色彩错配**。

### 3.3 形态语言问题

| 设计元素         | 当前                                 | 问题                                                                 |
| ---------------- | ------------------------------------ | -------------------------------------------------------------------- |
| **圆角**         | 1.5rem (卡片主力), 最高 2rem         | 大圆角传达亲和、柔和、孩童化；安全工具需要精确、锐利、不妥协的几何感 |
| **阴影层数**     | 4 层柔和阴影                         | 过度结构化；Swiss 风格强调边界而非阴影来定义空间                     |
| **表面区分**     | 背景/卡片/次级表面之间仅 2-6% 亮度差 | 层次感弱，视觉扁平但非刻意                                           |
| **无图标**       | 仅 Unicode 文字符号                  | 安全工具可能受益于 SVG 图标系统来增强功能识别                        |
| **暗色模式底色** | `hsl(40, 15%, 12%)` 橄榄棕           | 带有暖色调的暗色在安全语境中显得不够中性冷静                         |

### 3.4 暗色模式专项问题

```
当前暗色模式: #232018 (橄榄棕底)
理想暗色模式: 偏冷的深蓝黑

问题:
├── 暖色调在低光环境下可能产生"泥土"感
├── 陶土橙在暗色背景下偏亮 (#e06741)，对比度过强
├── 卡片与背景的暖色差异在暗色下更不明显
└── 整体缺少安全工具常有的"终端/控制台"冷静感
```

### 3.5 问题总结表

| 维度       | 问题严重度 | 描述                               |
| ---------- | ---------- | ---------------------------------- |
| 品牌一致性 | 🔴 严重    | "Snow" + 暖大地色 = 品牌撕裂       |
| 品类匹配   | 🔴 严重    | 安全工具需要冷色，不是暖橙         |
| 色心理     | 🟡 中等    | 陶土橙传达手工/餐饮/生活方式       |
| 圆角语言   | 🟡 中等    | 1.5-2rem 圆角过于柔和              |
| 暗色模式   | 🟡 中等    | 橄榄棕底 vs 安全工具期待的冷蓝暗色 |
| 阴影       | 🟢 轻微    | 4 层阴影偏重但影响有限             |
| 可读性     | 🟢 轻微    | 对比度基本满足 AA 标准             |
| 字体       | 🟢 优秀    | Inter + Fira Code 选择很好，保留   |

## 4. v2.2 新设计方案核心思想

### 4.1 设计哲学转向

```
v2.1:  Warm Earth-Tone  →  暖、有机、手工艺感、生活品牌
         │
         │  ＜＜＜  品类/品牌不匹配  重构  ＞＞＞
         │
v2.2:  Swiss Modernism 2.0 × Minimalism
       冷、精确、可信、安全工具
```

### 4.2 风格选择：为什么是 Swiss Modernism 2.0 + Minimalism

Swiss Modernism 2.0 和 Minimalism 是 UI UX Pro Max 框架中的两个风格。选择它们的理由与工具性质高度共振：

| 风格特征         | 对 Snow Cues 的意义                                    |
| ---------------- | ------------------------------------------------------ |
| **几何精确性**   | 密码工具需要不妥协的精确感；方形/轻微圆角表达"无误差"  |
| **信息层级清晰** | Swiss 风格擅长内容结构化；帮助用户在大量加密数据中导航 |
| **克制装饰**     | 安全工具不应有"装饰性"元素；每个视觉元素都有结构目的   |
| **高对比度**     | 文字清晰可读是信任的基础                               |
| **专业冷静**     | Swiss 风格被企业、银行、政府广泛信任                   |
| **无时效性**     | Swiss Modernism 不过时；密码工具不是追时髦的           |

**UI UX Pro Max 风格编号参考**：

- **1号** Minimalism & Swiss Style：企业应用、仪表盘、文档 — 非常适合工具类界面
- **32号** Swiss Modernism 2.0：企业站点、建筑、编辑 — 传达精确和专业信任

Snow Cues v2.2 取两者的交集：Minimalism 的内容层级 + Swiss Modernism 2.0 的几何精确性。

### 4.3 色彩选择：为什么是 Glacier Blue（冰川蓝）

| 设计依据       | 说明                                                                           |
| -------------- | ------------------------------------------------------------------------------ |
| **品牌直觉**   | "Snow" 直接指向冰、雪、冷的意象。冰川蓝天然契合                                |
| **色心理**     | 蓝色是安全、信任、技术、专业的最强色彩信号                                     |
| **品类惯例**   | 密码管理器/安全工具行业普遍使用蓝色系（1Password, Bitwarden, LastPass 均蓝色） |
| **对比度优势** | 中等蓝色在白色背景上有良好可读性                                               |
| **暗色适配**   | 蓝色在深色背景上比暖色更自然稳重                                               |
| **饱和度控制** | `#3E7CB1` 不是高饱和亮蓝，而是略带灰度的成熟蓝 — 避免"太跳"                    |

```
暖色系:  🔴🟠🟡  →  温暖、食欲、生活方式
冷色系:  🔵🔷  →  信任、安全、技术、专业

Snow Cues 定位 = 安全 + 技术 + 信任 = 蓝色系
```

### 4.4 圆角哲学：为什么从 1.5rem 降到 0.5rem

视觉心理学研究表明：

```
大圆角 (>1rem):  亲和、柔和、孩童化、消费品牌、社交媒体
小圆角 (<0.5rem): 精确、专业、技术、工业、工具
直角 (0):         硬派、严肃、工程师文化、终端

安全工具的最优区间: 0.375–0.75rem
```

具体设定：

- 按钮/输入框: `0.375rem` — 紧贴功能，微圆角缓冲手指触感
- 卡片/面板: `0.5rem` — 明显的矩形几何感，保持精确
- 最大: `0.75rem` — 仅用于最突出的卡片（如存储数据卡片）

**这个设定同时满足**：

1. Swiss Modernism 的精确几何感
2. 触摸设备友好的微圆角
3. 与安全/专业工具的品类期望一致

### 4.5 阴影哲学：从 4 层柔和到 3 层极简

```
v2.1 阴影: 多层叠加 + 超低透明度 = "柔和模糊感"
v2.2 阴影: 单层精确 + 略高透明度 = "清晰层级感"

Swiss 设计传统更依赖边框(而非阴影)定义空间。
阴影仅用来区分"浮起"的层面 — 有明确的层级目的。
```

### 4.6 暗色模式哲学：从暖橄榄到冷蓝黑

```
v2.1 暗色: #232018 = 暖橄榄棕
v2.2 暗色: #0F1724 = 深极夜蓝

差异:
├── v2.1: 暖调、泥泞感 → 像森林木屋的壁炉边
├── v2.2: 冷调、清澈感 → 像雪夜下的控制中心
└── 后者更符合 "Snow Cues" 的夜晚想象
```

### 4.7 设计 Token 完整性

v2.2 设计系统覆盖 v2.1 的所有 token 类别并扩展：

| Token 类别  | v2.1            | v2.2                 | 变化                           |
| ----------- | --------------- | -------------------- | ------------------------------ |
| 颜色 (亮色) | 13 个           | 17 个                | + `--primary-muted`, `--error` |
| 颜色 (暗色) | 13 个           | 17 个                | 同上                           |
| 圆角        | 6 级 (0.5–2rem) | 5 级 (0.375–0.75rem) | 收紧范围，移除过大值           |
| 阴影        | 4 级 (含 glow)  | 4 级 (含 glow)       | 结构不变，值简化               |

### 4.8 反模式意识

从 UI UX Pro Max 方法论借鉴了"反模式清单"概念 — 明确列出不该做的事情，这比只说"该做什么"更具操作性：

```
❌ AI 紫/粉渐变    → 安全工具不是 AI 产品
❌ 玻璃态/透明度     → 安全工具需要不透明、坚固
❌ 亮霓虹色        → 破坏专业冷静感
❌ 纯黑 #000       → 暗色模式用深蓝黑更护眼
❌ emoji 做图标    → SVG 图标更专业
❌ >300ms 动画     → 安全工具需要即时响应
❌ 硬编码颜色      → 必须走 token
❌ 无焦点态        → 无障碍的底线
❌ 大圆角          → 与 Swiss 几何哲学冲突
```

## 6. 设计决策记录 (Design Decision Records)

### DDR-001: 为什么全线切冷色而不是微调暖色？

**考虑过**：保留暖色系但降低饱和度、调整色调使其更中性。

**否决原因**：

1. "Snow" 品牌名是硬约束 — 任何暖色调都与"雪"存在根本张力
2. 安全品类的色彩惯例是蓝色系
3. 渐进调整会留下"既不够暖也不够冷"的尴尬中间态
4. 一次完整切换比多轮微调更彻底，也更省事

### DDR-002: 为什么选 Swiss Modernism 而不是其他风格？

**考虑过**：

- Glassmorphism：否决 — 透明/玻璃效果削弱安全工具的"坚固"感
- Neumorphism：否决 — 过于柔和，不适合数据密集型界面
- Brutalism：否决 — 太极端，不便于普通用户
- 保留 Warm Earth-Tone 只换色：否决 — 圆角/阴影也需要同步调整

**最终选择 Swiss Modernism 2.0 + Minimalism**：两者天然适合工具类/企业级产品，几何精确、信息层级清晰、装饰最小化。

### DDR-003: 为什么 Inter + Fira Code 保留不变？

1. Inter 是 Swiss 风格的自然搭配 — 几何无衬线体、高可读性、广泛支持
2. Fira Code 是密码/代码展示的最优选择 — 连字、清晰区分 `0`/`O`、`1`/`l`、`I`
3. 中文字体回退链（PingFang SC, Microsoft YaHei）已充分覆盖中文用户
4. 更换字体会带来较大的实施成本和可能的排版回归

### DDR-004: 为什么最大圆角定在 0.75rem？

因为 `1rem` 及以上圆角开始呈现"pill"感，与 Swiss 的 rectilinear 几何哲学冲突。0.75rem 是"明显能看到圆角但仍感觉是矩形"的最大值。

### DDR-005: 为什么暗色模式用 `#0F1724` 而不是 `#000`？

1. 纯黑 `#000` 在 OLED 上虽然有省电优势，但对比度过高导致眼部疲劳
2. 微蓝色调的暗色更符合"雪夜/极夜"的品牌意象
3. 从暖橄榄切换到冷蓝黑，在暗色模式下也能感受到品牌转向
4. 安全工具的"控制台感"用深蓝黑营造更自然

### DDR-006: 为什么建立本地 SVG 图标系统而不是引入图标库？

**考虑过**：直接引入 Lucide、Heroicons 或其他第三方图标库。

**最终选择本地 SVG React 组件**：

1. 当前真实需求只有主题切换、操作指引展开收起和空间品牌标记，完整图标库会增加不必要依赖。
2. 本地图标组件足以覆盖 v2.3 阶段 3 的范围，也符合项目不添加 UI 组件库的约束。
3. `currentColor` 让图标自然继承亮/暗主题 token，避免额外颜色规则。
4. 图标默认 `aria-hidden`，必须配合可见文字或 `aria-label`，避免用图标单独表达安全关键含义。
