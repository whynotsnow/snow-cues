# Snow Cues v2.3 开发计划

> 本文档是 v2.3 的分阶段实施总纲。每个阶段独立可交付、可单独派发、单独提交。
> 设计权威以 `design/Design.md` 为准；本计划解释「为什么这样排期」与「每阶段做什么」。

---

## 0. 背景与现状勘误

本次规划建立在一个必须先正视的事实之上：**`design/Design.md` 写的是 v2.2 Glacier Blue，但代码从未真正切换** —— Design.md 实际上是「设计意图文档」，而不是「已落地状态文档」。

已核实的差异（截至本计划撰写时）：

| 维度 | Design.md（v2.2 目标） | 代码实际（src/styles.css 等） | 差异 |
| --- | --- | --- | --- |
| `--primary` | `#3E7CB1` 冰川蓝 | `hsl(15,65%,52%)` 陶土橙 | 主色未切换 |
| 亮色 `--bg` | `#F7F8FA` | `hsl(38,28%,91%)` | 未切换 |
| 暗色 `--bg` | `#0F1724` | `hsl(40,15%,12%)` | 未切换 |
| `--radius-2xl` / `--radius-3xl` | 已删除 | `1.5rem` / `2rem` 仍在，`var(--radius-2xl)` 引用 30+ 处 | 未收紧 |
| `--error` token | 存在 | 完全缺失 | 缺失 |
| `--primary-muted` | 存在 | 缺失 | 缺失 |
| `--accent-warm` | 不存在 | 仍在（无业务引用） | 多余 |
| 语义 muted/border/fg 变体 | 未定义 | 未定义 | 双方都缺 |
| index.html theme-color | Glacier Blue | `#d25636` / `#3a2f24` | 未切换 |
| index.html Inter 字重 | `400;500;600;700` | 含 `wght@300` | 多了 300 |
| vite.config.ts 主题色 | Glacier Blue | `#d25636` / `#f3ece1` | 未切换 |
| design-system/README.md | 应描述 Glacier Blue | 仍整段描述 warm-earth-tone | 自相矛盾 |
| 图标 | 「不要 emoji，用 SVG」 | 无 SVG 系统；emoji 仅 3 处（ThemeToggle / NavRail / GuidanceDrawer） | 策略未落地 |
| 页面级设计规范 | 缺失 | 缺失 | 缺失 |

结论：v2.3 的首要工作不是「继续叠加新特性」，而是**先把 v2.2 设计系统真正落地到代码，消除文档与实现的撕裂**，再在其上叠加设计系统收口工作。

---

## 1. 范围与目标

v2.3 围绕「设计系统落地 + 收口」组织，共 6 个阶段（0–5）：

| 阶段 | 名称 | 性质 |
| --- | --- | --- |
| 0 | 文档对齐勘误 | 文档 |
| 1 | 落地 v2.2 Glacier Blue token 到代码 | 代码（核心） |
| 2 | 语义 token 完整化 | 代码 + 文档 |
| 3 | 最小 SVG 图标系统（DDR-006） | 代码 + 文档 |
| 4 | 页面级设计规范 | 文档 |
| 5 | 安全文案保护红线 | 文档 |

### 不在 v2.3 范围内（红线）

- 不引入 UI 组件库、不引入远程规则市场、不执行用户 JS。
- 不改变任何安全边界、存储格式、字段白名单、密钥生命周期。
- 阶段 1 的纯视觉迁移不得改动业务流程文案、按钮可用性与 handler 调用顺序（遵循 `docs/coding-rules.md`）。

## 3. 各阶段详细任务

### 阶段 0 · 文档对齐勘误

**目标**：消除「Design.md 说 Glacier Blue、README 说 warm-earth-tone」的自相矛盾，让文档先诚实描述现状。

**改动**：

- `src/ui/design-system/README.md`：删除 warm-earth-tone token 表与「warm-earth-tone（Anthropic 品牌灵感）」「legacy alias 保持 `--color-*` 兼容」等失实描述（代码中已无 `--color-*` / `legacy`）。改写为「当前 token 体系正在向 Glacier Blue 收敛，权威以 `design/Design.md` 为准」。
- `design/Design.md` §14：新增「实施状态备忘」，明确 v2.2 Glacier Blue 迁移**尚未落地**，v2.3 阶段 1 负责。避免后续 Agent 误以为 Design.md 已是现状。
- `design/design-review.md` 末尾：新增 v2.2 实施状态勘误一条。

**验收**：三个文档对「当前实现是什么颜色」描述一致；无代码改动。
**提交信息**：`docs: correct design system implementation status`

### 阶段 1 · 落地 v2.2 Glacier Blue token 到代码（核心阶段）

**目标**：把 Design.md 的 token 值真正写入代码。这是 v2.2 本应做但没做的迁移。**纯视觉 token 替换，零安全边界改动**。

**改动（精确到文件）**：

1. **`src/styles.css` 颜色 token**（`:root` 与 `:root[data-theme="dark"]`）：
   - 亮色：按 Design.md §3.1 写入全部值（`--bg #F7F8FA` / `--fg #1A1D23` / `--card #FFFFFF` / `--card-fg #1A1D23` / `--muted #EEF1F5` / `--muted-fg #5B6370` / `--popover #FFFFFF` / `--border #DDE1E6` / `--input #DDE1E6` / `--primary #3E7CB1` / `--primary-fg #FFFFFF` / `--primary-strong #2F6492` / `--primary-muted rgba(62,124,177,0.08)` / `--ring #3E7CB1`）。
   - 暗色：按 Design.md §3.2 写入。
   - 语义色：`--success` / `--warning` / `--info` 两套值；**新增 `--error`**（亮 `#C74B4B`、暗 `#D96363`）。
   - **删除 `--accent-warm`**（已核实无业务引用）。

2. **`src/styles.css` 圆角**：
   - `:root` 调整 `--radius: 0.5rem`、`--radius-sm: 0.375rem`、`--radius-md: 0.5rem`、`--radius-xl: 0.625rem`，新增上限 `--radius-2xl` 退役说明与 `--radius-pill: 999px`。
   - **删除 `--radius-2xl`(1.5rem) 与 `--radius-3xl`(2rem)**。
   - 全文 `var(--radius-2xl)` → `var(--radius)`（30+ 处）。
   - 规则：新代码不得使用 > `--radius-xl`(0.75rem) 的圆角。

3. **`src/styles.css` 阴影**：4 层柔和 → Design.md §7 的 3 层极简（`--shadow-sm/md/lg` + `--glow-primary` 冰川蓝 `rgba(62,124,177,0.2)`）。

4. **`src/styles.css` 新增规范**：`--duration-fast 150ms` / `--duration-normal 200ms` / `--duration-slow 300ms`；`space-1..6` 间距 scale（CSS 变量可选，沿用现有直写值模式）。

5. **`index.html`**：theme-color 改 `#3E7CB1`(light) / `#0F1724`(dark)；Inter 字重 `400;500;600;700`（去掉 `300`）。

6. **`vite.config.ts`**：`THEME_COLOR = "#3E7CB1"`、`BACKGROUND_COLOR = "#F7F8FA"`；manifest 注释更新。

7. **图标重生成**：更新 `public/icons/icon.svg`（陶土橙底 → 冰川蓝底），运行 `npm run generate-icons` 重生 192/512/maskable/apple-touch/favicon。

**验证（最小受影响）**：
- `npm run typecheck`（token 替换不改类型，预期通过）
- `npm run build`（含 PWA manifest 主题色校验）
- UI 测试不动前置条件/断言，预期通过；若担心只跑受视觉影响的少量用例。
- 视觉核对由用户亲自做（AGENTS.md：不启动 dev/preview）。

**安全边界核对**：仅替换视觉 token；未改 `PasswordEntry` / `PasswordGroup` / `spaceProfiles` / `migrationBatches` 字段、未改 session/key 生命周期、未改 storageData 格式、未改任何 handler。

**提交信息**：`feat(design): land v2.2 glacier blue design tokens`

### 阶段 2 · 语义 token 完整化

**目标**：把状态色扩展成完整语义矩阵，杜绝 CSS 里散写 `rgba(...)`。

**新增 token 矩阵**（写入 `:root` 与 `[data-theme="dark"]`）：

| 语义 | `--*-muted`（浅底） | `--*-border`（描边） | `--*-fg`（文字） |
| --- | --- | --- | --- |
| success | `rgba(45,138,94,0.08)` | `rgba(45,138,94,0.18)` | `#1E5E40` |
| warning | `rgba(194,132,45,0.07)` | `rgba(194,132,45,0.22)` | `#6B4A1E` |
| error | `rgba(199,75,75,0.06)` | `rgba(199,75,75,0.2)` | `#6B2E2E` |
| info | `rgba(75,127,199,0.06)` | `rgba(75,127,199,0.18)` | `#2D4A78` |

暗色模式对应提高背景透明度、文字提亮，参照 Design.md §3.3 的 notice 表。

**收口动作**：
- `.notice-info/.notice-success/.notice-warning/.notice-error` 把硬编码 `rgba(...)` 改为 `var(--info-muted)` 等。
- badge 的 dirty/clean/deprecated 配色改为 `var(--warning-muted)` 等。
- 全文检索剩余 `rgba(` 与 hex 直写，收敛到 token。

**Design.md §3 增补**：新增「语义 token 矩阵」小节，记录上表。

**验收**：grep 不再有业务性 `rgba(语义色,...)` 直写；通知/badge 在亮暗模式下颜色一致；typecheck/test/build 通过。
**提交信息**：`feat(design): complete semantic token matrix`

### 阶段 3 · 最小 SVG 图标系统（DDR-006）

**目标**：建立本地 SVG React 图标组件，替换现有 emoji/Unicode。

**新增目录**：`src/ui/icons/`（纯展示组件，不依赖 controller/crypto/storage）。

**首批图标**（6–10 个，按真实使用裁剪）：

- `SunIcon` / `MoonIcon` → 替换 ThemeToggle `☀️🌙`。
- `CollapseIcon` / `ExpandIcon`（或 `ChevronLeft/Right`）→ 替换 GuidanceDrawer `◀▶`。
- `SpaceIcon`（品牌/空间入口用，替换 NavRail `◈`）。
- 备选：`KeyIcon`、`ShieldIcon`、`CopyIcon`、`CheckIcon`（供后续密码/状态页按需引入）。

**图标组件规范**（写入 Design.md 新 §9.x 与 DDR-006）：

- 纯 SVG，`fill="currentColor"` / `stroke="currentColor"`，颜色继承父级。
- 通过 `size` prop 支持 `16 | 18 | 20`（默认 18）。
- 不承载敏感含义；**任何图标都必须配合可见文字标签**，不单独用图标表达操作。
- 无 emoji、无 Unicode 符号（落实 Design.md 反模式 #6）。
- 组件只做展示，不含业务逻辑（遵守 design-system 分层规则）。

**替换动作**：仅替换上述 3 处现有 emoji/Unicode；业务页图标化留待后续按页迁移（不在 v2.3 强制范围）。

**Design.md 增补**：新增 §9.x「图标系统」规范；在 design-review.md 新增 **DDR-006: 图标系统选择**（不引入 UI 组件库、优先本地 SVG React 组件、只维护少量安全工具必要图标、16/18/20px、currentColor、图标不承载敏感含义且必须有文字标签配合）。

**验收**：3 处 emoji/Unicode 全部替换；新图标在亮/暗模式下颜色随 `color` 正确变化；可访问性：图标有 `aria-hidden`，操作按钮的文字标签仍在；typecheck/test/build 通过。
**提交信息**：`feat(ui): add minimal SVG icon system`

### 阶段 4 · 页面级设计规范

**目标**：补齐 Design.md 当前只有「App Shell」、缺「页面级」规范的空白。

**Design.md 新增章节「页面级设计规范」**，覆盖：

- **页面骨架清单**：空间主页、规则管理、输出适配、密码管理、系统工具 5 个 hash 页面各自的典型结构（SectionHeader + 卡片组顺序、页面级通知区 `.page-notice-area` 与主体的间距规则）。
- **主列与表单宽度约束**：把现有 `main-column max-width 1050px`、表单 label 560px / textarea 760px / 校验选择器 560px / 解密密钥 `calc(50% - 0.375rem)`（Design.md §10.3）落到页面级说明。
- **SectionHeader 页面级用法**：每页一个主标题 + 描述 + trailing actions 的统一模式。
- **移动端（960px 断点）页面折叠规则**：多列网格 → 单列、Guidance 抽屉静态全宽、storageData 卡片全宽。
- **空状态与步骤指引的页面级位置**：`.empty-state`、`Steps` 在列表页的统一摆放。

**实施**：本阶段主要是 Design.md 文档增补；代码侧的页面级规范落地留待后续按页迁移任务。

**提交信息**：`docs(design): add page-level layout spec`

### 阶段 5 · 安全文案保护红线

**目标**：Swiss Minimalism 的「少装饰多内容」容易诱导 Agent 删文案；Snow Cues 安全模型复杂，必须立红线。

**新增条款**（写入 Design.md 设计原则 + AGENTS.md 关键路径 + docs/coding-rules.md + docs/product-rules.md）：

> **安全关键流程不得为了视觉极简删除必要解释。** 以下概念在「首次出现」或「关键操作前」必须保留中文说明：
> - `storageData`（存储数据）
> - `master_password`（空间主密码）
> - `entrySecret` / 关键密钥
> - `encrypted_memory_hint` / 记忆提示
> - 空间校验
> - 规则链
> - 输出适配
> - 迁移模式
> - 历史空间
> - 归档空间

**依据**：核实 `src/ui/guidance.ts` 已为这些概念编写了完整的中文说明，现状良好；本红线是为了**防止后续极简化重构误删**。

**落地位置**：

- Design.md：在「设计原则」表后新增「安全文案保护」原则，并在反模式表新增一条「为视觉极简删除安全关键文案」。
- coding-rules.md：新增「UI 文案删减约束」编码边界。
- product-rules.md：在「UI 状态与导航」补一句安全文案保留要求。
- AGENTS.md：在「后续开发红线」补一条。

**提交信息**：`docs: add security-critical copy protection rule`
